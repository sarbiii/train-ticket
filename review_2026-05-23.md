# ticket_web.py 코드 리뷰

- 리뷰일: 2026-05-23
- 대상 파일: `ticket_web.py`
- 검증 커맨드: `python3 -m py_compile ticket_web.py`
- 결론: 현재 기능은 동작 가능한 단일 사용자용 스크립트에 가깝지만, 외부 접속 웹 서버로 운영하기에는 인증, 상태 격리, 입력 검증, 스나이프 조회 범위에서 먼저 고쳐야 할 위험이 있습니다.

## 주요 발견사항

### P0. 외부 접속 서버인데 인증/접근 제어가 없습니다

- 위치: `ticket_web.py:88`, `ticket_web.py:679-683`
- 근거: Flask 앱이 `host="0.0.0.0"`로 뜨고, 안내 문구도 Tailscale 외부 접속을 전제로 합니다. 그런데 `/api/search`, `/api/snipe/start`, `/api/snipe/stop`, `/api/state`에 인증이나 토큰 검증이 없습니다.
- 영향: 같은 네트워크/Tailscale 접근자가 검색, 스나이핑 시작/중단, 예약 성공 정보 조회를 할 수 있습니다. `/api/state`는 선택 열차, 로그, 예약 성공 정보까지 노출합니다.
- 개선:
  - 최소한 `WEB_AUTH_TOKEN` 환경변수 기반 Bearer 토큰 또는 고정 헤더 allowlist를 추가합니다.
  - Tailscale만 믿지 말고 앱 레벨 인증을 둡니다.
  - `/api/state`는 인증 없이는 절대 반환하지 않도록 합니다.

### P0. 최초 검색에서 고른 뒤쪽 열차를 스나이프 루프가 못 볼 수 있습니다

- 위치: 검색 전체 수집 `ticket_web.py:198-240`, `ticket_web.py:242-279`; 스나이프 루프 단일 조회 `ticket_web.py:503-514`
- 근거: `/api/search`는 `from_time`부터 `to_time`까지 페이지네이션으로 여러 배치를 수집합니다. 반면 `_snipe_thread`는 매 루프마다 `client.search_train(dep, arr, date, from_time, ...)` 한 번만 호출하고, 그 결과 안에서 `train_ids`를 찾습니다.
- 영향: 사용자가 조회 결과에서 뒤쪽 시간대 열차를 선택해도 스나이프 루프의 첫 배치에 없으면 `find_train_by_id`가 계속 `None`이 되어 감시 대상에서 빠집니다. 로그상으로는 정상 반복처럼 보이므로 운영자가 놓치기 쉽습니다.
- 개선:
  - `_snipe_thread`에서도 `_search_all_trains` / `_search_all_srt_trains`를 사용해 최초 검색과 같은 범위를 조회합니다.
  - `to_time`을 `snipe/start` payload에 포함하고 필수 검증합니다.
  - 선택 열차를 찾지 못한 경우 `train_statuses`에 누락 경고를 남깁니다.

### P1. 전역 `_session` 하나에 모든 사용자/탭 상태가 섞입니다

- 위치: `ticket_web.py:91-104`, `ticket_web.py:313-344`, `ticket_web.py:366-389`, `ticket_web.py:427-437`
- 근거: 검색 결과, 로그인 client, 스레드, 로그, 성공 정보가 전역 딕셔너리 하나에 저장됩니다.
- 영향: 다른 브라우저나 다른 탭에서 검색하면 기존 사용자의 client/search/snipe 상태가 덮입니다. 새로고침 복원은 되지만 "한 명만 쓰는 서버"라는 강한 전제가 생깁니다.
- 개선:
  - 최소 구현: 단일 사용자 모드임을 명시하고 동시 접근 차단 락을 둡니다.
  - 운영 구현: `session_id`를 발급해 브라우저별 상태를 분리합니다.
  - client 객체와 thread 상태는 같은 세션 단위로만 참조하게 합니다.

### P1. `api_snipe_start` 입력 검증이 부족해 스레드가 조용히 죽을 수 있습니다

- 위치: `ticket_web.py:366-389`, `ticket_web.py:441-450`, `ticket_web.py:496`
- 근거: `data["dep"]`, `data["arr"]`, `data["date"]`, `data["from_time"]`를 바로 인덱싱하고, `interval`은 바로 `float()` 변환합니다. `train_type=srt`인데 SRTrain import가 실패한 상태에서도 start 요청은 막지 않습니다.
- 영향: 잘못된 요청이 들어오면 Flask 응답은 `"started"`가 될 수 있지만, 백그라운드 스레드가 `KeyError`, `ValueError`, `NameError`로 죽고 사용자에게 원인이 전달되지 않습니다.
- 개선:
  - `api_snipe_start`에서 필수 필드, 날짜/시간 형식, `train_type`, `seat_option`, `interval`, `train_ids` 형식을 검증합니다.
  - SRT 미설치 상태에서 `train_type=srt` 요청은 400/500으로 즉시 거절합니다.
  - `_snipe_thread` 최상단에 broad exception guard를 두고 `push("error", msg=...)` 후 종료합니다.

### P1. 검색/상태 변경 시 전역 client와 thread 접근이 락 없이 섞입니다

- 위치: `_session` 쓰기 `ticket_web.py:313-344`, `ticket_web.py:368-388`; 상태 락 일부 `ticket_web.py:126-157`
- 근거: 상태 이력은 `_state_lock`으로 보호하지만, `client`, `thread`, `log_queue`, `stop_flag`, `train_type` 쓰기는 락 없이 진행됩니다.
- 영향: 검색 요청과 스나이프 시작/중단 요청이 동시에 오면 스레드가 다른 train_type의 client를 받거나, stop flag와 queue가 엇갈릴 수 있습니다.
- 개선:
  - `_session` 접근을 전담하는 작은 상태 관리 함수로 모으고 동일 락을 사용합니다.
  - `api_snipe_start`는 현재 search params와 payload train_type이 일치하는지 확인합니다.
  - client 객체를 전역 최신값이 아니라 `snipe` payload에 묶인 세션 상태에서 가져옵니다.

### P1. `/api/search`가 실제 오류를 "열차 없음"처럼 숨깁니다

- 위치: `ticket_web.py:198-214`, `ticket_web.py:242-252`, `ticket_web.py:315-347`
- 근거: `_search_all_trains`와 `_search_all_srt_trains`가 `except Exception: break`로 모든 예외를 삼킵니다. 그 결과 상위 API는 `in_range`가 비었다고 판단해 404 "열차가 없습니다"를 반환합니다.
- 영향: 로그인 만료, 네트워크 오류, 외부 API 응답 변경, 파싱 오류가 모두 "열차 없음"으로 보입니다. 장애 판단이 어려워집니다.
- 개선:
  - `NoResultsError`와 네트워크/로그인/파싱 오류를 구분합니다.
  - 알 수 없는 예외는 로그에 남기고 502/500으로 반환합니다.
  - 사용자 메시지는 간단히, 서버 로그는 원인과 traceback을 남깁니다.

### P2. SSE가 단일 queue 소비 모델이라 여러 탭/재연결에서 실시간 이벤트가 나뉠 수 있습니다

- 위치: `ticket_web.py:391-410`
- 근거: `/api/snipe/stream` 접속자들이 같은 `queue.Queue`에서 `get()`을 호출합니다. Queue는 fan-out이 아니라 경쟁 소비입니다.
- 영향: 탭이 2개 이상 열리거나 새로고침 직전 기존 EventSource가 잠깐 살아 있으면 이벤트 일부가 한쪽 연결에만 전달될 수 있습니다. 상태 이력 저장으로 상당 부분 완화됐지만 실시간 표시에는 여전히 빈틈이 있습니다.
- 개선:
  - 이벤트를 append-only history로 저장하고 각 SSE 연결은 cursor 이후 이벤트를 읽게 합니다.
  - 간단한 구현은 `Condition` + event list + index 방식이 적합합니다.

### P2. 수동 중단 이벤트가 상태 이력에 즉시 기록되지 않습니다

- 위치: `ticket_web.py:412-420`, `ticket_web.py:654`
- 근거: `api_snipe_stop`은 queue에 `{"type": "done"}`을 직접 넣지만 `_record_event`를 호출하지 않습니다. 실제 스레드가 깨어나 `push("done")`을 실행해야 이력에 남습니다.
- 영향: 중단 직후 새로고침하면 `/api/state`의 logs에 종료 이벤트가 없을 수 있습니다. 반대로 스트림에는 done이 먼저 도착해 종료된 것처럼 보입니다.
- 개선:
  - stop API에서도 `push`와 같은 경로로 이벤트를 기록합니다.
  - 스레드 종료 done과 사용자 중단 done을 구분해 중복 표시를 피합니다.

### P2. 상태는 메모리 한정이라 tmux/프로세스 재시작 시 복원되지 않습니다

- 위치: `ticket_web.py:91-104`, `ticket_web.py:679-683`
- 근거: 상태가 전역 dict에만 저장됩니다.
- 영향: 새로고침은 복원되지만 서버 재시작, tmux 세션 종료, 코드 재배포 후에는 상태가 사라집니다.
- 개선:
  - 단기: "서버 프로세스가 살아있는 동안만 복원"을 UI에 명확히 표시합니다.
  - 장기: SQLite 또는 JSONL에 상태/이벤트 이력을 저장합니다.

### P2. 포트, tmux 세션명, host가 하드코딩되어 운영 유연성이 낮습니다

- 위치: `ticket_web.py:72`, `ticket_web.py:679-683`
- 근거: `ktx-web`, `8188`, `0.0.0.0`이 코드에 고정되어 있습니다.
- 영향: 여러 인스턴스 실행, 로컬 전용 실행, systemd/docker 배포 시 코드 수정이 필요합니다.
- 개선:
  - `WEB_HOST`, `WEB_PORT`, `TMUX_SESSION_NAME`, `DISABLE_TMUX` 환경변수를 지원합니다.
  - 기본값은 지금과 같게 유지해 기존 실행법을 깨지 않게 합니다.

### P2. Discord 알림 제목이 SRT 성공에도 KTX로 고정될 수 있습니다

- 위치: `ticket_web.py:174`, `ticket_web.py:183`
- 근거: `_discord_notify` embed title/footer가 KTX/Korail 기준 문구입니다. payload의 `train_type`은 이미 존재합니다.
- 영향: SRT 예약 성공 시 알림 문구가 부정확합니다.
- 개선:
  - `train_type = res.get("train_type", "KTX")`를 기준으로 title/footer를 분기합니다.

### P3. 입력 형식 검증과 에러 메시지 표준화가 필요합니다

- 위치: `ticket_web.py:282-364`, `ticket_web.py:441-450`
- 근거: 날짜는 `YYYYMMDD`, 시간은 `HHMMSS`를 기대하지만 길이/숫자/범위 검증이 없습니다.
- 영향: 잘못된 입력이 외부 예매 API까지 전달되고, 실패 원인이 사용자에게 명확히 나오지 않을 수 있습니다.
- 개선:
  - `validate_search_payload`, `validate_snipe_payload`를 분리합니다.
  - 날짜/시간 파싱은 `datetime.strptime`으로 검증합니다.
  - 오류 응답 스키마를 `{ "error": "...", "code": "..." }` 형태로 통일합니다.

## 권장 개선 순서

1. 인증 추가: 모든 `/api/*`에 토큰 검증 적용.
2. 스나이프 조회 범위 수정: 최초 검색과 동일한 `to_time` 범위를 반복 조회.
3. `api_snipe_start` payload 검증과 스레드 최상위 예외 처리 추가.
4. `_session` 접근을 락 기반 상태 관리 함수로 정리.
5. SSE queue를 이벤트 히스토리 fan-out 구조로 변경.
6. 운영 옵션을 환경변수화.

## 최소 테스트 제안

```bash
python3 -m py_compile ticket_web.py
python3 - <<'PY'
import ticket_web

client = ticket_web.app.test_client()
resp = client.get('/api/state')
assert resp.status_code == 200
data = resp.get_json()
assert 'running' in data
assert 'search' in data
print('api/state ok')
PY
```

추가로 mocking 가능한 구조로 바꾼 뒤에는 다음 시나리오를 테스트해야 합니다.

- 검색 결과 2페이지 이후 열차를 선택했을 때 스나이프 루프가 해당 열차를 계속 감시하는지.
- `train_type=srt`인데 SRTrain이 없을 때 `/api/snipe/start`가 즉시 실패하는지.
- `/api/snipe/stop` 직후 새로고침해도 종료 상태가 복원되는지.
- 인증 토큰이 없거나 틀린 요청이 401/403으로 차단되는지.

## 운영/보안 체크리스트

- `WEB_AUTH_TOKEN` 같은 앱 레벨 인증 필수.
- `.env` 권한은 600 유지.
- Tailscale ACL로 접근 가능한 계정/디바이스 제한.
- 예약 성공 정보는 `/api/state`에서 필요한 필드만 최소 노출.
- 서버 재시작 시 상태가 사라지는 점을 운영 문서에 명시.
- 외부 API 오류는 "열차 없음"과 분리해 로그로 남김.
